# План: Исправление проблемы с кириллицей при экспорте

## Проблема

При экспорте диаграммы через кнопку "Export" в студии кириллица отображается некорректно.

## Корневая причина

**Проблема в `electron/python-bridge.ts:179`:**

```typescript
let json = JSON.stringify(message) + '\n';
json = json.replace(/[\ud800-\udfff]/g, '');  // ← ПРОБЛЕМА: удаляет ВСЕ символы Unicode >= 0xD800
this.process.stdin.write(json, 'utf8');
```

**Почему это ломает кириллицу:**
1. Кириллические символы находятся в диапазоне `U+0400` - `U+04FF` (и расширенные диапазоны)
2. Функция `replace(/[\ud800-\udfff]/g, '')` должна удалять **только** недопустимые UTF-16 суррогаты (половинки суррогатных пар)
3. Однако, если строка была неправильно закодирована/декодирована ранее, символы кириллицы могут быть искажены
4. В JavaScript/TypeScript строки всегда UTF-16, поэтому кириллица (`U+0410` = 'А') должна обрабатываться корректно

## Где ещё возможны проблемы

### 1. Python-side: `sys.stdin.readline()` без явной кодировки

**Файл:** `packages/core/src/rpaforge/bridge/server.py:163`

```python
def _read_line(self) -> str:
    """Blocking read line for executor."""
    try:
        return sys.stdin.readline()  # ← Использует системную кодировку на Windows (cp1251 на русской Windows)
    except Exception:
        return ""
```

**На Windows:**
- `sys.stdin.readline()` использует `sys.stdin.encoding` 
- По умолчанию это `cp1251` (Windows-1251) на русской Windows
- Electron отправляет UTF-8, а Python ожидает системную кодировку → **MISMATCH**

### 2. Client-side codegen уже содержит workaround

**Файл:** `packages/studio/src/stores/processStore.ts:190-213`

```typescript
if (sanitized.includes('Р') || sanitized.includes('С')) {
  // Попытка исправить кодировку кириллицы
  // Это указывает на то, что проблема уже была замечена ранее
}
```

## Решение

### Шаг 1: Исправить Python-side кодировку stdin

**Файл:** `packages/core/src/rpaforge/bridge/server.py:160-165`

Изменить:

```python
def _read_line(self) -> str:
    """Blocking read line for executor."""
    try:
        line = sys.stdin.buffer.readline()  # Читаем байты, не строку
        return line.decode('utf-8')  # Явно декодируем UTF-8
    except Exception:
        return ""
```

**Обоснование:**
- `sys.stdin.buffer` даёт доступ к сырым байтам
- Явная декодировка в UTF-8 гарантирует правильную интерпретацию данных
- Electron всегда отправляет UTF-8, так что это соответствует протоколу

### Шаг 2: Убрать ненужную санитизацию UTF-16 суррогатов

**Файл:** `packages/studio/electron/python-bridge.ts:178-180`

Изменить:

```typescript
let json = JSON.stringify(message) + '\n';
// Убрать: json = json.replace(/[\ud800-\udfff]/g, '');
this.process.stdin.write(json, 'utf8');
```

**Обоснование:**
- `JSON.stringify()` уже корректно кодирует Unicode, включая кириллицу
- Удаление суррогатов было попыткой исправить проблему, но маскирует реальную проблему на Python-side
- UTF-16 суррогаты - это валидная часть UTF-16, если они в правильных парах

### Шаг 3: Проверить и исправить sanitizeString функции

**Файлы:**
- `packages/studio/src/utils/clientCodegen.ts:10-12`
- `packages/studio/src/domain/codegen/generator.ts:10-12`
- `packages/core/src/rpaforge/codegen/generator.py:10-16`

Текущая реализация:

```typescript
function sanitizeString(str: string): string {
  return str.replace(/[\uD800-\uDFFF]/g, '');
}
```

Это **правильно** - удаляются только непарные суррогаты (invalid standalone surrogates). 

Оставить как есть.

### Шаг 4 (опционально): Добавить тесты на кириллицу

Создать тесты, которые проверяют:
1. Генерацию кода с кириллическими названиями процессов
2. Генерацию кода с кириллическими строками в аргументах
3. Круговой путь: диаграмма → JSON → экспорт → валидация кириллицы

## Порядок исправления

1. **Сначала исправить Python-side** (шаг 1) - это основная проблема
2. **Затем убрать workaround в Electron** (шаг 2)
3. **Проверить работу** с кириллическими данными
4. **Добавить тесты** (шаг 4)

## Ожидаемый результат

После исправления:
- Кириллица в названиях процессов, условиях, сообщениях корректно экспортируется
- JSON-RPC протокол правильно передаёт Unicode символы между Electron и Python
- Нет необходимости в дополнительных workaround на клиентской стороне

## Риски

- Низкий риск: изменение кодировки stdin может повлиять на существующие данные
- Митигация: UTF-8 - стандарт для JSON-RPC, так что это скорее исправление бага, чем breaking change

## Дополнительные замечания

1. **stdout тоже нужно проверить** - Python отправляет ответы через `sys.stdout.write()`, но там уже используется `json.dumps()` который по умолчанию UTF-8, так что проблем нет.

2. **Windows-специфичность** - проблема проявляется только на Windows, так как на Unix системах `sys.stdin` обычно UTF-8 по умолчанию.

3. **ProcessStore workaround** - существующий код в `processStore.ts:190-213` можно удалить после исправления основной проблемы.
